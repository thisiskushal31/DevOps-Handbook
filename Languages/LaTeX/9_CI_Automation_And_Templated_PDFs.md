# CI, automation, and templated PDFs

[← Back to LaTeX](./README.md)

LaTeX fits well into automated workflows: the source is plain text, so it can be version-controlled, generated by scripts, and compiled in CI. This topic covers running LaTeX in continuous integration, scripting builds, and producing PDFs from templates (e.g. filled with data from another system).

## LaTeX in continuous integration

Goals of LaTeX in CI:

- **Build on every change:** Compile the document on push or pull request so breakage is caught early.
- **Reproducible artifacts:** Produce the same PDF from the same source and same TeX distribution.
- **Optional:** Publish the PDF as a build artifact or to a release page.

CI runners need a TeX distribution and the packages your document uses. Two common approaches:

1. **Install TeX on the runner:** On Ubuntu, install a TeX Live metapackage (e.g. `texlive-full` or `texlive-latex-extra` and `texlive-fonts-recommended`) so `pdflatex` and `latexmk` are available. Then run `latexmk -pdf main.tex` (or your chosen engine and main file).
2. **Docker image:** Use an image that already contains TeX Live (e.g. `texlive/texlive` or a slimmer variant). This gives consistent versions and avoids long installs on each run.

Example (GitHub Actions, Ubuntu runner, install then build):

```yaml
- name: Install TeX Live
  run: |
    sudo apt-get update
    sudo apt-get install -y texlive-latex-extra latexmk

- name: Build PDF
  run: latexmk -pdf -interaction=nonstopmode main.tex

- name: Upload PDF
  uses: actions/upload-artifact@v4
  with:
    name: document
    path: main.pdf
```

Use `-interaction=nonstopmode` so the build does not stop for prompts. For complex projects, use the same engine and options as locally (e.g. `-pdflatex=xelatex` if you use XeLaTeX).

## Templated PDFs from data

When the content of the report is driven by data (e.g. from a database, API, or script), you can:

1. **Generate .tex from a template:** A script (Python, Ruby, etc.) reads data and fills a template (e.g. Jinja2, ERB, or simple string substitution) to produce a `.tex` file, then run the TeX engine to get a PDF.
2. **Keep a fixed .tex and pass variables:** Some setups use a small number of placeholders and substitute them at build time (e.g. version number, date, environment name).

Template example (concept): the template has placeholders like `{{ title }}` and `{{ sections }}`; the script replaces them with real title and with generated `\section{...}` blocks, then runs `pdflatex` on the result. The script is responsible for escaping special characters (e.g. `\`, `{`, `}`) if the data is user-supplied, to avoid injection.

## Security and robustness in automation

- **Untrusted input:** If any part of the content comes from users or external systems, sanitize or escape it before inserting into `.tex`. LaTeX has many special characters (`\`, `{`, `}`, `#`, `$`, `&`, `_`, `%`, `^`) and powerful commands; unescaped input can change the structure of the document, trigger file inclusion (`\input`, `\include`), or run external commands in some setups. Escape or sanitize all user-supplied or external data, or use a minimal templating layer that only substitutes safe placeholders.
- **Reproducible builds:** Pin the TeX distribution (e.g. Docker image tag or specific package versions) so the same commit produces the same PDF. Lockfiles or fixed image digests help. Avoid relying on “install latest” in CI if you need bit-identical output.
- **Secrets:** Do not put passwords or API keys in the repository. Use CI secrets or environment variables and inject them only when needed (e.g. for publishing), not into the document source unless necessary and safe.
- **Build isolation:** Running the TeX engine on untrusted `.tex` can be risky if the document or class files can execute arbitrary code. For high-assurance pipelines, use a dedicated build environment and avoid loading packages or running helpers from untrusted locations.

## When to automate

Automate LaTeX builds when:

- The document is in a repository and you want every change to be buildable.
- Reports are generated from live or batch data and need to be regenerated often.
- You want a single command or pipeline to produce the final PDF for release or distribution.

Using latexmk (or a single script that runs the correct engine and bibliography steps) keeps the automation simple and consistent with local development.
