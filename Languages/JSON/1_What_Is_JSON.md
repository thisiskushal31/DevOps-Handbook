# What is JSON?

JSON (JavaScript Object Notation) is a lightweight, text-based data interchange format. It is language-independent: the same JSON text can be produced and consumed by programs written in JavaScript, Python, Java, Go, or any language with a JSON library. It is easy for humans to read and write and easy for machines to parse and generate. That combination makes JSON the default format for APIs, configuration files, and pipeline definitions across software engineering, DevOps, and cloud-native systems.

**What it is.** JSON is built on two universal data structures: a **collection of name/value pairs** (realized in most languages as an object, record, or map) and an **ordered list of values** (realized as an array, list, or sequence). In JSON these appear as objects `{ "key": value }` and arrays `[ value, ... ]`. Values can be strings, numbers, the literals `true`, `false`, or `null`, or nested objects and arrays. There are no comments, no trailing commas, and no single-quoted strings—only the minimal grammar needed for interchange. The format is an Internet Standard and is aligned with international standards so that parsers and generators are interoperable across platforms and languages.

**Why it is used.** JSON replaced XML in many contexts because it is simpler, smaller, and maps directly to native data structures in most programming languages. REST and GraphQL APIs typically use JSON for request and response bodies. CI/CD configs (e.g. GitHub Actions, GitLab CI), cloud provider APIs (AWS, GCP, Azure), and infrastructure-as-code outputs (Terraform state, provider config) use JSON. Package manifests like `package.json` (Node.js/npm) and many tool configs are JSON. In security and observability, logs and policy definitions are often JSON. Because it is ubiquitous, understanding JSON is essential for reading configs, debugging API payloads, and writing automation.

**How you use it.** You either **produce** JSON (serialize in-memory data to a string or file) or **consume** it (parse a string or file into native data structures). Every major language has a standard or de facto standard library for JSON. You do not “run” JSON; you parse it and then act on the resulting data. Validation (e.g. with JSON Schema) is separate from parsing and is used to enforce structure and types when the data comes from untrusted or external sources. Implementation in a given stack—how to parse or generate JSON in JavaScript, Python, CI pipelines, or Terraform—lives in the handbook sections for those technologies (e.g. Languages/JavaScript, CiCd, IAC). This JSON section is the place to learn how JSON itself works: grammar, semantics, edge cases, validation, and format-level security.

**Use cases.** JSON appears wherever structured data must be exchanged or stored as text: API requests and responses, configuration files, CI/CD pipeline definitions, Terraform state and provider config, Kubernetes manifests (when not YAML), `package.json` and lockfiles, log and event payloads, and security/audit outputs. For using JSON in a specific tool or language, see the corresponding handbook folder (CiCd, IAC, Cloud-Native, Languages/JavaScript, etc.).

**Data interchange and text-based.** "Data interchange" means one system produces a representation of data (names, numbers, lists, nested structure) and another system consumes it. The representation must be unambiguous and agreed upon—that is what the JSON grammar provides. "Text-based" means the format is a sequence of characters (encoded as UTF-8 bytes), not binary. You can open a JSON file in a text editor, read it, and write it. That makes debugging and version control straightforward; the tradeoff is size and parse cost compared with binary formats.

**One document, one value.** A valid JSON text is exactly one value. That value can be an object (most common for configs and API payloads), an array (common for lists of items), or a primitive (string, number, or literal). There is no way to have two or more top-level values in one JSON text; formats like JSON Lines use one value per line to represent multiple values. Understanding this rule helps you interpret parser errors and choose the right structure.

**Why JSON succeeded.** JSON is minimal: a small set of rules, no schema required to parse, and no namespaces or attributes. It maps directly to the native data structures of almost every language (objects/maps and arrays/lists). That makes serialization and deserialization trivial and predictable. By contrast, XML has tags, attributes, namespaces, and mixed content; parsing and mapping to in-memory structures is heavier. For APIs and config, JSON’s simplicity won: less payload size, less parser complexity, and direct use in JavaScript (browsers and Node). The tradeoff is that JSON has no built-in schema, comments, or binary types; those are handled by conventions (e.g. JSON Schema, JSONC, or string/Base64 for binary).

**When to choose JSON.** Use JSON when you need a single, widely supported text format for structured data: APIs (REST, GraphQL), config files that tools and editors understand, state or logs that must be readable and diffable, and any boundary between systems or languages where both sides can parse it. Prefer strict JSON at boundaries (APIs, files exchanged between tools) so every consumer behaves the same. Use variants (JSON5, JSONC) only for human-edited config in a controlled pipeline that strips or converts before interchange. For very large or high-throughput binary payloads, consider binary encodings (e.g. MessagePack, BSON) while keeping the logical model JSON-like; the Advanced topic in this section covers when that matters.
